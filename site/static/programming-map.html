STEP 1

環境構築 — プロジェクトの土台を組む💡概念📘用語

📌 ポイント: 上の src/ を支えるのは OS → Node → npm → .env の 4 層。下の層が崩れると上は動かない。
Node.js ★ JavaScript を PC 上で動かすエンジン
使う場面npm/Vite/Next.js すべての前提。最初の1回 nvm install 20。覚え方: 「ブラウザの外で JS を動かす VM」
VSCode ★ 定番の統合開発環境（IDE）
使う場面本リポは VSCode 前提。clients.code-workspace を開けば全社が同時起動する。覚え方: 「家＝IDE」
ターミナル ★ npm を打つ画面
使う場面Mac は Terminal、Windows は PowerShell。VSCode 内蔵ターミナル(Ctrl+`)が便利。
npm ★ ライブラリを取ってくるコマンド
使う場面npm install で package.json 通りに一括取得。同等品: pnpm / yarn — 速度や省スペース重視で乗り換え可
package.json ★ プロジェクトの設計書
中身名前 / 依存ライブラリ / scripts(npm run dev など)。触ったらコミット必須。本リポでは set-ports.js が scripts.dev を自動書換するので注意。
node_modules ○ 取得したライブラリの置き場
注意巨大なので .gitignore に必ず入れる。直接触らない。バグ調査でいじっても再 install で消える。
package-lock.json △ 依存バージョン固定
使う場面「自分の PC では動くのに本番で動かない」を防ぐ。commit する。
.env ★ 秘密の設定ファイル
絶対ルールAPI キー / パスワード / Supabase キー等を入れる。絶対 git に含めない。.env.example で書式だけ共有する。
.gitignore ★ Git に含めないファイルを宣言
必須項目node_modules/ .env* dist/ .next/。事故を1回でも起こすと信用ゼロになる箇所
.env.example △ 値を空にしたサンプル
運用新メンバーがクローンしてすぐ動かせるように、必要キーの一覧だけ git に入れる。
NEXT_PUBLIC_* ★ ブラウザに公開してOKな env
厳格な区別サーバー専用キー(SERVICE_ROLE 等)を NEXT_PUBLIC_ に絶対しない。漏洩事故の典型。

STEP 1-2

フレームワーク・ライブラリ — 何で作るか💡概念📘用語

📌 ポイント: React が中心。それを SSR/SEO 込みで包んだのが Next.js、軽量 SPA 用が Vite。下に TS + Tailwind の共通足回り。
Next.js ★ 本サイト・業務システムの第一選択
強みページ・ルーティング・サーバー処理まで込み。SEO・SSR・Image最適化が標準。本リポの全社で採用率トップ
Vite ★ 開発が爆速なビルドツール
強み起動0.3秒・HMR(差分更新)が一瞬。SPA・管理画面・社内ツールに最適。SEOが要らない領域はこちら
Nuxt ○ Vue ベースの Next.js 相当
使い分けVue を採用しているプロジェクトでのみ。本リポでは原則 React 寄せ。
Angular ○ Google 製の大規模向け
位置づけ業務系大規模で TypeScript 強制。本リポは特定 1 社のみ。
React ★ UI を「部品」で作る発明
核心「画面を関数で書く」発想。Next.js も Vite も中身は React。「コンポーネント = 1つの部品 = 1つの関数」
shadcn/ui △ コピペで導入する UI 部品集
強みnpm install でなく コードを直接コピーするので所有権が手元に残る。Tailwind 前提。
Radix UI ○ 中身はあるがスタイル無しの土台
位置づけshadcn/ui の中身。アクセシビリティ完備のヘッドレス UI。
TypeScript ★ 型チェックで事故を事前に潰す
効果「undefined のメソッドを呼んだ」を保存時に発見。大規模ほど効く。本リポは新規プロジェクト原則 TS
Tailwind CSS ★ クラス名で書く CSS
強み命名地獄から解放、デザインの一貫性。bg-blue-500 px-4 のように属性で組む。
CSS Modules △ クラス名衝突を自動回避
使い分けTailwind が合わないチーム/旧コードベースで採用される伝統的選択肢。

STEP 2-3

開発中の作業 — 1日の作業ループ💡概念📘用語

npm run dev ★ 開発サーバー起動
使う場面毎日最初に叩くコマンド。package.json の scripts.dev が呼ばれる。本リポは set-ports.js がポートを自動割当
ホットリロード(HMR) ★ 保存で即反映
仕組み差分だけ再ロード。状態を保ったまま見た目だけ変えられる。Vite が特に速い
ポート番号 ○ 複数サーバー同時起動の入口
本リポの規則クライアント = 3001〜3009 / エージェント生成 = 4001〜。物理分離で衝突回避。
ビルド △ 本番用にまとめ・圧縮
使う場面npm run build。デプロイ前にローカルで通る確認。Vercel は push で自動実行。
DevTools ★ ブラウザ内蔵の開発者ツール
主要タブElements(DOM) / Console(JS) / Network(通信) / Application(localStorage)。F12 で開く。
console.log ★ 値を覗く最速デバッグ
コツconsole.log({user, posts}) のようにラベル付きで複数値を一度に出す。commit前に消す
デバッガー △ ブレークポイントで停止
使い分けconsole で追えない複雑な状態は VSCode の Run and Debug を使う。Next.js は next dev --inspect。
Network タブ ★ API 通信を可視化
使う場面「データが画面に出ない」のとき真っ先に見る。Status / Response / Headers を確認。
ESLint ★ コードのバグ検出
役割「未使用変数」「any 型」「危険な書き方」を保存時に警告。npm run lint。
Prettier ★ コードの自動整形
役割インデント・改行・カンマを統一。保存時自動に設定するのが標準。
リファクタリング △ 動きを変えず整理
原則テストが通ったまま構造を改善。3回似たコードを書いたら共通化を検討。
定数一元管理 ★ URL/電話/LINEID は1ファイルに
本リポルールconfig/constants.ts に集約。コンポーネント直書き禁止。ドメイン変更時に1箇所触れば済む。
VSCode ★ 定番のエディタ
必須拡張ESLint / Prettier / GitLens / Tailwind IntelliSense / Claude Code。本リポは clients.code-workspace 起動が前提
Claude Code 拡張 ★ エディタ内 AI 補助
位置づけVSCode 上の UI ラッパー。実体は CLI/SDK。詳しくは Section B の「IDE関係」参照。
GitLens △ 行ごとの blame を表示
使い場面「この行誰が書いた？なんで？」が即わかる。コードレビュー時に強力。

データベース・バックエンド — Supabase 一式で揃える📘用語

Supabase ★ 本リポの推奨バックエンド
強みPostgres + Auth + Storage + RLS + Realtime が一式。本リポは1プロジェクトに複数アプリをスキーマ分離で相乗り運用
Firebase ○ Google 製の旧来 BaaS
方針本リポでは新規拡張しない。残存機能のみ維持し、新規は Supabase へ寄せる。
D1 △ Cloudflare の SQLite 互換
使い場面LINE Webhook の会話履歴など、軽量・Edge 完結な用途に限定。本リポでは LINE-CRM 系で利用。
Postgres ★ 業界標準の RDB
強みSQL 標準対応・拡張機能が豊富(pgvector で AI 検索も可)・どこでも動く可搬性。
マイグレーション ★ スキーマ変更を履歴管理
使う場面本番テーブル変更は必ず migration ファイル経由。Supabase MCP の apply_migration で打てる。
mapper 層 ★ DB↔TS の名前変換
本リポ規則lib/supabase-mappers.ts。生データを画面で直接使わない。型崩れ防止と保守性。
ORM ○ SQL の代わりにコードで DB 操作
選択肢Prisma / Drizzle / Kysely。本リポでは Supabase の TypeScript SDK で直接書く方が多い。
RLS（行レベル制御） ★ DB 側で「誰が見られるか」
使い方auth.uid() = user_id のような policy を SQL で書く。アプリ側のチェックを忘れても DB が止める最後の砦。
Supabase Auth ★ ログイン基盤
機能メール / OAuth(Google/GitHub) / Magic Link を SDK 1 行で。JWT を返してくれるので RLS と直結
JWT △ 認証トークンの標準形式
仕組み「ユーザーID + 期限 + 署名」を Base64 エンコード。Supabase Auth が裏で発行・検証している。
service_role ★ RLS を無視できる強権限キー
絶対ルールサーバー側だけで使う。ブラウザに漏らしたら詰む。NEXT_PUBLIC_ 接頭辞絶対禁止。
REST ★ URL で操作する標準形式
仕組みGET /api/posts POST /api/posts。Supabase は PostgREST で自動生成してくれる。
GraphQL △ 1 エンドポイントで欲しい列だけ取る
強み過剰取得を防げる。弱みキャッシュ設計が REST より難しい。本リポでは Payload v3 で標準同梱。
Server Action ★ Next.js 13+ の API レス書き方
強み'use server' 関数を直接呼ぶ。型が貫通し API ルート定義不要。Vercel での標準
Realtime △ Supabase の WebSocket 配信
使い場面チャット / 通知 / 共同編集。テーブル変更を購読すると勝手に push される。

SEO・LLMO — Google にも AI 検索にも拾われる📘用語

メタタグ ★ 検索結果と SNS シェアの見た目
必須<title> <meta description> og:image。Next.js なら generateMetadata() で。
sitemap.xml ★ 全URLの一覧
役割「ここに何があるか」を Google に通告。次の Web サービスでも標準。
robots.txt △ クローラーへの指示
使い方Disallow: /admin 等で巡回禁止区域を指定。機密の隠し場所ではない（公開ファイルなので）。
canonical △ 重複URLの正本宣言
使う場面同じ記事に複数 URL がある時(?utm=... 付き等) 正本だけ評価させる。
OGP ★ SNS シェア時のカード
効果X / LINE / Slack で URL を貼った時の見栄え。og:title og:image og:description の3つは必ず。
JSON-LD ★ 構造化データの記法
位置<script type="application/ld+json"> で <head> に埋め込む。
BlogPosting ★ 記事ページ用スキーマ
必須項目headline author publisher datePublished。本リポはブログ記事に必ず付ける。
FAQPage ★ Q&A を構造化
LLMO効果AI 検索の引用元になりやすい。Google 検索ではアコーディオン表示。
Speakable △ 音声検索の読み上げ対象
効果AI 検索や音声アシスタントが「ここを読み上げて」を判断する目印。cssSelector で h1/サマリーを指定。
HowTo ○ 手順記事用スキーマ
使い場面「Aの作り方」「Bの直し方」記事に付けると Google でステップ表示される。
Core Web Vitals ★ Google 重視の体感速度指標
3指標LCP(最大要素表示) / INP(操作応答) / CLS(レイアウトずれ)。悪いと検索順位が下がる。
Lighthouse ★ 速度・SEO 採点
使い方Chrome DevTools 内蔵。本リポは Chrome DevTools MCP 経由でも回せる。
Image Optimization ★ 画像を WebP/AVIF へ自動変換
Vercelnext/image が画面サイズ別に配信。LCP 改善の最大の武器。

STEP 2-6

セキュリティ — 4 つの攻撃面を塞ぐ💡概念📘用語

HTTPS ★ 通信の暗号化
本リポVercel/Render は自動で有効化。src/** に http:// 直書き禁止(ミックスコンテンツ事故防止のため check-no-http.mjs でビルドガード)。
CORS △ 別ドメイン通信の許可制御
仕組みブラウザが Origin ヘッダで判断。サーバー側で Access-Control-Allow-Origin を返して許可する。
WAF △ Webアプリ用ファイアウォール
本リポCloudflare DNS 経由で WAF 二段防御を被せる方針。Vercel Pro 以上は Vercel Firewall も。
レート制限 △ 同一 IP からの連投を絞る
使い場面ログイン試行・LINE Webhook の連投・AI API。Cloudflare の Rate Limiting Rule で実装が手軽。
API キー管理 ★ 公開鍵 / 秘密鍵を厳密に分離
絶対ルールSupabase の service_role は絶対に NEXT_PUBLIC_ に置かない。漏洩は即詰みなので環境変数の命名規則で物理的に分ける。
JWT ★ 認証トークンの標準形式
仕組み「ユーザーID + 期限 + 署名」を Base64 エンコード。Supabase Auth が内部で使う。
RLS（行レベル制御） ★ DB 側で誰が見られるか制御
本リポアプリ側のチェックを忘れても DB が止める最後の砦。本リポはほぼ全テーブルに RLS 入り。
セッション管理 △ Cookie / localStorage の使い分け
原則機密トークンは httpOnly Cookie へ。localStorage は XSS で抜かれる可能性あり。
XSS 対策 ★ スクリプト埋込攻撃を防ぐ
仕組みReact/Next.js は {value} で自動エスケープ。危険: dangerouslySetInnerHTML を信頼できない入力に使うと突破される。
SQL インジェクション ★ DB に不正な SQL を混ぜる攻撃
防御プレースホルダ/ORM/Supabase SDK を使えば自動回避。文字列連結で SQL を作らない。
CSRF ○ なりすましリクエスト攻撃
防御Cookie + トークン検証 / SameSite 属性。Server Action 利用時は Next.js が自動で対策。
入力バリデーション ★ 受信値を必ず検証
標準Zod / Valibot で型と中身をスキーマ検証。クライアント側だけの検証は信用しない。
Dependabot ★ 脆弱性検知して自動 PR
使う場面GitHub Settings → Security でON。npm の CVE が出たら自動で更新 PR を作る。
npm audit △ CLI で脆弱性チェック
使い方npm audit fix で自動修正可能なものは即対処。CI の最後に必ず通すと安心。
lockfile 維持 ★ バージョン固定で再現性確保
原則package-lock.json を必ずコミット。CI/本番と手元の挙動を一致させる。
Secret Scanning △ 漏洩キーを自動検知
仕組みGitHub が push 時に AWS/Stripe 等のキー形式を検知して通知。誤って .env を commit した時の最後の砦。

STEP 7

パフォーマンス・運用監視 — 公開後の生存確認📘用語

ログ ★ サーバーの動作記録
使う場面「なぜか動かない」の最初に見る場所。Vercel/Cloudflare のダッシュボードか CLI/MCP で取得。
エラー監視 ★ 本番エラーを自動収集
定番Sentry。スタックトレース・発生頻度・どのユーザーかをまとめてくれる。
構造化ログ △ JSON 形式で出す
強み後で集計しやすい。console.log({event:"login", userId, ms}) のように key:value で。
アナリティクス ★ ユーザー行動の計測
選択肢GA4(無料・複雑) / Plausible(有料・シンプル・Cookie 不要)。本リポはケースバイケース。
イベント計測 △ ボタンクリック等を個別記録

SLA / SLO ○ 稼働率の目標値

方針既存プロジェクトで Firebase が残っていても新規機能は Supabaseに寄せる。二重管理を避けるため Firebase は拡張しない。

選定理由静的/Node/Python を同一プラットフォームで扱え、日本からの近さで Singapore リージョン固定。無料枠あり。

役割分担CI(検証)= Actions / CD(配信)= Render のpush連携。responsibility split で問題切り分けが楽。
定数一元化 ★ URL/電話/LINEは config/constants.ts に集約
なぜドメイン変更・LINE ID変更時に1ファイルだけ触れば全箇所反映。コンポーネント内ハードコード禁止。
HTTPS強制 ★ src/** に http:// 直書き禁止
事故防止Render等のHTTPS サイトからhttp://画像を読むとブラウザがサイレントブロック。ローカルでは気付かず本番で画像全滅する典型事故を防ぐ。proxyImage() / upgrade-insecure-requests / check-no-http.mjs の三重ガード
Supabase mapper層 ★ snake↔camel変換を必ず挟む
ルールlib/supabase-mappers.ts で変換。生データを直接画面で使うと型不整合と保守性低下を招く。
JSON-LD ★ ブログ個別ページは BlogPosting+FAQPage+Speakable
LLMOFAQPage と Speakable はAI検索最適化に効く。コンテンツサイトでは標準装備。
環境変数フォールバック △ NEXT_PUBLIC_SITE_URL ?? SITE_URL_CONST
使い方Render側で本番URLを切替しつつ、ローカル開発では定数のデフォルトが効く。両環境で壊れない書き方。

理由Unsplashの画像IDは削除されることがあり、404画像が本番で見えると見栄えが壊れる。使う前の確認が必須。
